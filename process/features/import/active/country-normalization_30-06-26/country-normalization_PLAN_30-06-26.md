# Country Normalization — Push Ingest + TSV Import

Complexity: SIMPLE
Status: VALIDATED — Gate: PASS (2026-06-30); ready for EXECUTE
Date: 30-06-26
Feature: import

> **TL;DR.** Adds `normalizeCountry()` to `import-utils.ts`, wires it into both data-entry
> paths (TSV import + scraper push ingest), adds an indexed `country` column to `crm_leads`,
> and updates the filter/dropdown queries to use that column — so "Philippines" and "Singapore"
> filter reliably regardless of what raw text the scraper or TSV export sends.

---

## Overview

Country data enters the CRM from two paths:

| Path | Where country originates | Current behaviour |
|------|--------------------------|-------------------|
| TSV import (`scripts/import.ts`) | `venue_country` column (col 32) merged with `venue_city` → stored in `crm_leads.location` as `"City, Country"` | Raw, un-normalized |
| Push ingest (`/api/leads/ingest`) | `location` field in JSON payload | Raw string from scraper, no normalization |

The existing filter (`listLeadsFiltered`) does exact-match on `location`.
Because `location` combines city + country as free text, "Philippines" never matches
"Makati City, Philippines" — and variant spellings ("PH", "philippines", "Pilipinas")
break the match entirely.

**Fix:** Add a dedicated `country` text column populated from a `normalizeCountry()` utility
during every write. Filter and dropdown queries switch to the `country` column.

---

## Goals

- G1. `normalizeCountry()` maps known variant spellings → canonical names
  ("PH", "Pilipinas", "the Philippines" → "Philippines"; "SG", "Singapura" → "Singapore").
- G2. Every new lead written via TSV import or push ingest gets a populated `country`.
- G3. The `country` filter (`listLeadsFiltered`) and dropdown (`getLeadCountries`) use the
  new column, so "Philippines" and "Singapore" work as first-class filter values.
- G4. No breaking change to the ingest API wire format — `country` is derived server-side
  from either the existing `location` field or from `venue_country` in the TSV.

---

## Scope

**In scope:**
- `normalizeCountry()` utility in `src/lib/server/import-utils.ts`
- Additive `country` column on `crm_leads` (nullable, indexed) + Drizzle migration
- TSV import: populate `country` from `normalizeCountry(venue_country)`
- Push ingest: populate `country` from `normalizeCountry()` applied to the raw `location`
  field (parse last segment after the final comma, or full string if no comma)
- `getLeadCountries()` — switch to `selectDistinct({ country })` on `country` column
- `listLeadsFiltered()` — switch country-filter clause to `eq(crmLeads.country, country)`
- Unit tests for `normalizeCountry()` covering Philippines, Singapore, and null return for unknowns
- Historical backfill: existing `country = NULL` rows are backfilled by re-pushing scraper events through the ingest endpoint, which upserts `country` when it is `NULL` and an incoming location yields a PH/SG match (`scripts/push-from-scraper.ts`)

**Out of scope (v1):**
- Backfilling country for already-imported leads (data that already exists will have `country = NULL` until re-imported or manually updated)
- Other country mappings beyond PH and SG (can add later as new markets appear)
- City-level normalization
- Changing the `location` field format — it stays as-is for human display

## Implementation Checklist

### Step 1 — `normalizeCountry()` utility

- [ ] In `src/lib/server/import-utils.ts`, add:

```ts
const COUNTRY_MAP: Record<string, string> = {
  // Philippines
  'philippines': 'Philippines',
  'ph': 'Philippines',
  'pilipinas': 'Philippines',
  'the philippines': 'Philippines',
  'republic of the philippines': 'Philippines',
  'phil': 'Philippines',
  'phils': 'Philippines',
  'rp': 'Philippines',
  // Singapore
  'singapore': 'Singapore',
  'sg': 'Singapore',
  'singapura': 'Singapore',
  'republic of singapore': 'Singapore',
};

export function normalizeCountry(raw?: string | null): string | null {
  if (!raw) return null;
  const key = raw.trim().toLowerCase();
  return COUNTRY_MAP[key] ?? null;  // unknown inputs return null — no passthrough
}
```

- [ ] Export `normalizeCountry` from `src/lib/server/import-utils.ts`
- [ ] Mirror export in `scripts/lib/import-utils.ts` (the CLI re-export)

### Step 2 — Schema migration

- [ ] Run `bun run db:generate` to produce the migration file

### Step 3 — TSV import

### Step 4 — Push ingest

- [ ] In `src/routes/api/leads/ingest/+server.ts`, add a `parseCountryFromLocation()` helper (pure function, no import side-effects):

```ts
function parseCountryFromLocation(location?: string | null): string | null {
  if (!location) return null;
  const parts = location.split(',');
  return parts.length > 1 ? parts[parts.length - 1].trim() : location.trim();
}
```

- [ ] Apply: `country: normalizeCountry(parseCountryFromLocation(lead.location))` in the insert
- [ ] Import `normalizeCountry` from `$lib/server/import-utils`

### Step 5 — DB query layer

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Scraper sends `location` as `"Philippines"` only (no city prefix) — `parseCountryFromLocation` returns the whole string which is then normalized correctly | No action needed — function handles single-segment strings |
| Scraper sends `"Makati, Metro Manila, Philippines"` (multi-comma) — `split(',').at(-1)` correctly takes the last segment | `parts[parts.length - 1]` handles this correctly |
| `getLeadCountries()` callers broken by shape change | Only one caller: `+page.server.ts`. Covered in Step 5. |
| Existing leads have `country = NULL` after migration | Acceptable for v1 — future re-imports or manual updates populate it. Add a `## Known Gaps` note. |
| TSV import re-export needed in scripts/lib | `scripts/lib/import-utils.ts` uses **explicit named exports** (not a wildcard). Execute agent MUST explicitly add `normalizeCountry` to the named export list — it does NOT pick up new exports automatically. |
